# APA 要件定義

## 1. 概要

### 1.1 プロダクト名

- APA: AI Powered AWS CLI

### 1.2 解決したい課題

- AWS CLI は強力だが、サービスごとのサブコマンドやオプションが多く、必要な構文を毎回思い出すのが難しい。
- そのため、ユーザーは Web ブラウザ上の AI に自然言語で質問し、返ってきたコマンドをターミナルにコピペして実行することが多い。
- この運用には、コンテキストスイッチ、コピペの手間、危険なコマンドをそのまま実行してしまう不安がある。

### 1.3 プロダクトビジョン

- APA は、AWS 操作に特化した CLI ネイティブの AI エージェントである。
- ユーザーの自然言語入力から適切な AWS CLI コマンドを組み立て、実行前に意図と影響範囲を分かりやすく提示し、安全に実行する。
- 目指す体験は「ターミナルの中に AWS CLI に詳しい相棒がいる」状態であり、同時に高リスク操作では必ず人間が主導権を持つ。

## 2. ゴールと非ゴール

### 2.1 ゴール

- ターミナル上で自然言語の要望から AWS CLI コマンドを生成できること。
- 適切なサービス、サブコマンド、フラグ、profile、region、query を含む AWS CLI コマンドを提案できること。
- ブラウザに移動せず、そのままターミナル内で確認と実行まで完結できること。
- 読み取り系と更新系を判別し、危険度に応じて確認やブロックを行うこと。
- 生成したコマンドとその理由を示し、ユーザーが AWS CLI の学習にも使えること。
- LLM 依存部分を分離し、将来的にモデル提供元を差し替えられること。

### 2.2 非ゴール

- AWS CLI 自体を置き換えること。
- 任意のシェルコマンドを実行する汎用エージェントになること。
- Terraform や CloudFormation のような IaC を全面的に置き換えること。
- IAM や既存認証の制約を回避すること。
- ユーザー確認なしで破壊的操作を自律実行すること。

## 3. 想定ユーザー

### 3.1 主対象

- AWS をたまに触る開発者
- AWS の概念は分かるが CLI 構文の暗記はしたくない SRE / Platform Engineer
- 調査系コマンドを素早く実行したい運用担当者

### 3.2 前提条件

- ターミナル操作に抵抗がないこと
- AWS CLI v2 がローカルにインストール済みであること
- profile、SSO、環境変数など、通常の AWS CLI の認証方法で AWS に接続できること

## 4. ユースケース

### 4.1 読み取り系

- 「東京リージョンで稼働中の EC2 インスタンスを一覧表示して」
- 「この profile で S3 バケットを一覧表示して」
- 「CloudWatch Logs から直近 100 行だけ見せて」
- 「billing を含む Lambda 関数を探して」

### 4.2 条件付き取得

- 「staging 環境の ECS サービスだけ表示して」
- 「直近 24 時間で失敗した Step Functions 実行を見つけて」
- 「ap-northeast-1 の ALB を名前順で一覧にして」

### 4.3 更新系

- 「この Lambda のメモリを 1024 MB に変更して」
- 「prod の ECS service web を 2 台にして」
- 「この SSM パラメータを更新して」

### 4.4 複数ステップ

- 「この ALB のターゲットグループを調べて、そのヘルスチェック設定を見せて」
- 「最新の ECS タスク定義 ARN を取得して、その内容を表示して」

## 5. プロダクト原則

### 5.1 Safety First

- 高リスク操作は黙って実行しない。
- 破壊的操作や広範囲操作では、意図と影響範囲を表示し、ユーザー確認を必須にする。

### 5.2 AWS 特化

- 汎用エージェントではなく、AWS CLI とその周辺体験に最適化する。

### 5.3 Terminal Native

- 入力、確認、実行、結果確認までをターミナル内で完結させる。

### 5.4 Transparency

- 何をするつもりか、どの profile / region に対して実行するか、どの AWS CLI コマンドを使うかを常に見える形にする。

## 6. スコープ

### 6.1 MVP スコープ

- `apa "..."` 形式で自然言語入力を受け取れる
- 自然言語から 1 つの AWS CLI コマンド、または短いコマンド列を生成できる
- profile と region を認識し、表示できる
- 実行前に構造化されたプランを表示できる
- コマンドを安全度で分類できる
- 読み取り系は条件付きで自動実行、更新系は明示確認の後に実行できる
- 実行結果の stdout / stderr を表示できる
- 生成理由を簡潔に説明できる
- 実行履歴をローカルに残せる

### 6.2 MVP で優先する AWS サービス

- STS
- EC2
- S3
- ECS
- Lambda
- CloudWatch Logs
- SSM Parameter Store

### 6.3 MVP で扱わないもの

- 任意のシェルパイプやリダイレクトを含むコマンド生成
- 長いワークフローオーケストレーション
- ブラウザログインを伴う認証フローの自動化
- IAM 権限昇格やクロスアカウント制御の代行

### 6.4 Post-MVP

- ターミナル内での対話的な追加質問
- 失敗したコマンドの自動修正提案
- 「その cluster」「さっきの profile」のようなセッション記憶
- チーム向けポリシー制御
- AWS ドキュメントや社内 Runbook の参照

## 7. 機能要件

### FR-1 自然言語入力

- システムは CLI から自由文の自然言語入力を受け付けること。
- MVP では日本語と英語を受け付けること。
- リソース名、ARN、ID、タグ値などは可能な限りそのまま保持すること。

### FR-2 AWS コンテキスト検出

- 利用可能な AWS profile を検出できること。
- 実効 region を AWS CLI 設定、環境変数、CLI オプションから判定できること。
- 実行に必要な context が不足している場合は、その不足内容をユーザーに示せること。

### FR-3 意図解析

- 入力文から AWS サービス、操作種別、対象リソース、絞り込み条件、出力形式を抽出できること。
- 読み取り系、更新系、破壊系を区別できること。
- 曖昧さが大きい場合は危険な推測をせず、確認または失敗を返すこと。

### FR-4 コマンドプランニング

- 実行前に構造化されたプランを生成すること。
- プランには少なくとも以下を含むこと。
  - 意図の要約
  - 対象サービス / API
  - 生成した AWS CLI 引数列
  - 前提や仮定
  - リスクレベル
  - 確認要否
- 実行器はシェル文字列ではなく引数配列ベースで AWS CLI を呼び出せること。

### FR-5 プレビュー表示

- 実行前に生成コマンドを表示できること。
- どの profile / region で実行するかを表示できること。
- 必要に応じて、重要フラグの意味を簡潔に説明できること。

### FR-6 確認とポリシー

- 読み取り系かつ曖昧性が低い場合のみ、確認なしで実行できること。
- 更新系コマンドでは必ず確認を要求すること。
- 削除、terminate、force、ワイルドカード対象などの高リスク操作は、より強い確認または実行拒否の対象にできること。
- すべての実行に確認を強制する設定を持てること。

### FR-7 実行

- ローカルの `aws` バイナリを使ってコマンド実行できること。
- stdout / stderr をターミナルに表示できること。
- コマンド終了コードを保持できること。
- AWS CLI 未導入、認証失敗、認可失敗では安全にエラー終了できること。

### FR-8 エラー処理

- AWS CLI のエラーを平易な文で要約できること。
- 可能な範囲で、設定不足や入力ミスに対する修正案を出せること。
- 更新系や破壊系を、ユーザー確認なしで自動再試行しないこと。

### FR-9 出力形式

- AWS CLI の生出力をそのまま表示できること。
- よくある読み取り系操作では、人が読みやすいサマリ表示を選べること。
- JSON など機械可読な出力形式を要求された場合は尊重できること。

### FR-10 監査性

- 実行ごとにローカル履歴を残せること。
- 履歴には timestamp、自然言語入力、生成コマンド、リスク判定、確認結果、終了コードを含めること。
- セキュリティ要件に応じて履歴保存を無効化できること。

### FR-11 設定

- 少なくとも以下を設定可能にすること。
  - デフォルトの model provider
  - 確認ポリシー
  - 出力スタイル
  - region / profile の既定動作
  - 履歴保存の有無と保持期間
- スクリプト用途の non-interactive モードを持てること。ただし安全制約は強めにすること。

### FR-12 LLM 抽象化

- LLM 呼び出し層と AWS CLI 実行層を分離すること。
- 将来 provider を差し替えても executor や policy を大きく変えずに済むこと。

## 8. 安全性要件

### 8.1 実行面の制約

- MVP では `aws` コマンド以外を実行しないこと。
- モデルが生成したパイプ、サブシェル、リダイレクト、`sh -c` 形式は受け付けないこと。

### 8.2 リスク分類

- `low`: 読み取り系かつ対象が比較的限定される操作
- `medium`: 既知の特定リソースに対する更新
- `high`: 削除、停止、terminate、置換、広範囲対象、対象不明瞭な操作

### 8.3 ガードレール

- 生成プランは構造化スキーマで検証すること。
- 実行前に `aws` から始まる正当な引数列かどうか検証すること。
- 未解決プレースホルダ、曖昧な対象、空引数を含む場合は実行拒否すること。
- モデル出力の構造化に失敗した場合は安全側に倒して失敗扱いにすること。

### 8.4 人間の主導権

- 高リスク操作では、ユーザーがキャンセルできる機会を必ず与えること。
- 生コマンドだけでなく、「何が起きるか」の要約を併記すること。

## 9. 非機能要件

### NFR-1 正確性

- MVP 対象シナリオのベンチマークにおいて、初回で妥当な AWS CLI コマンドを生成できる率を 90 パーセント以上とする。

### NFR-2 応答速度

- 単純な 1 コマンド要求では、通常ネットワーク条件下で 5 秒以内にプラン表示を開始できること。

### NFR-3 一貫性

- モデル出力が揺れても、確認フローと実行ポリシーの挙動は決定的であること。

### NFR-4 可搬性

- MVP は macOS と Linux で動作すること。

### NFR-5 セキュリティ

- AWS 認証情報は既存の AWS CLI の仕組みに委ねること。
- ログや LLM プロンプトに秘匿情報を不必要に含めないこと。

### NFR-6 観測性

- デバッグ時に構造化ログを有効化できること。
- ただしデフォルトでは秘密情報を記録しないこと。

### NFR-7 配布・依存関係

- `apa` コマンドはバイナリ単体で実行できること。事前に何らかのランタイムや依存ライブラリ、依存パッケージのインストールは可能な限り不要とする。追加の依存関係は `aws` コマンドのみとする。（ただし `aws` コマンドがインストール済みであることは強制しない。`plan` だけ実行したいユースケースを考慮する）
- 各環境向けにビルド済みのバイナリを生成して配布できること。

## 10. 想定アーキテクチャ

### 10.1 主要コンポーネント

- CLI frontend
- config loader
- AWS context detector
- planner
- model provider adapter
- policy engine
- executor
- history / logging module

### 10.2 Planner の出力スキーマ

- 少なくとも以下のフィールドを持つことを想定する。
  - `intent_summary`
  - `risk_level`
  - `requires_confirmation`
  - `aws_cli_args`
  - `profile`
  - `region`
  - `assumptions`
  - `explanation`

### 10.3 実行フロー

- 自然言語入力を planner に渡す
- planner の構造化出力をスキーマ検証する
- policy engine で安全性判定する
- 必要なら確認プロンプトを表示する
- executor が `aws` 引数列を実行する
- 結果と履歴を記録する

## 11. CLI UX 要件

### 11.1 基本コマンド

- `apa "<natural language request>"`
- `apa plan "<natural language request>"`
- `apa exec "<natural language request>"`
- `apa config ...`
- `apa history`

### 11.2 期待挙動

- `apa` は、安全な読み取り系であればプラン表示後にそのまま実行してよい。
- `apa plan` は生成コマンドを表示するだけで実行しない。
- `apa exec` は明示的な実行意思を示すサブコマンドだが、それでも確認ポリシーは無視しない。

### 11.3 UX 例

```text
$ apa "prod の ECS service web を 2 台にして"

Intent: ECS service "web" in cluster "prod" の desired count を 2 に変更する
Risk: medium
Profile: team-prod
Region: ap-northeast-1
Command:
  aws ecs update-service --cluster prod --service web --desired-count 2

This will change the running service capacity.
Execute? [y/N]:
```

## 12. 受け入れ基準

### 12.1 MVP 完了条件

- ユーザーがローカル環境に APA を導入して起動できること。
- 自然言語入力に対し、構造化プランを返せること。
- 安全な読み取り系要求を、ターミナルから end-to-end で実行できること。
- 更新系要求では、実行前に確認が必ず入ること。
- 曖昧または危険な要求では、危険な推測をせず明確に失敗できること。

### 12.2 テストカテゴリ

- 読み取り系リソース一覧
- region / profile 指定付きクエリ
- 特定リソースに対する更新操作
- 曖昧な入力に対する確認 / 拒否
- 存在しないリソース ID
- AWS 認証失敗 / 認可失敗
- 不正な model 出力

### 12.3 成功指標

- 代表プロンプト集に対する初回成功率
- 実行前キャンセル率とその理由
- 実行後エラー率
- ユーザーがブラウザに戻る回数の削減

- 初期リリースで更新系コマンドをどこまで許可するか

## 14. 推奨方針

### 14.1 MVP の安全境界

- 読み取り系は自動実行可
- 更新系は明示確認後に実行可
- 破壊系は MVP では原則ブロック、または `--dangerously-allow-destructive` のような強い opt-in が必要

### 14.2 初期対応サービス

- まずは EC2、S3、ECS、Lambda、CloudWatch Logs、SSM に絞る
- ここは AWS 利用頻度と CLI 活用頻度が高く、価値を出しやすい

### 14.3 技術方針

- planner は構造化 JSON を返す
- executor はローカルバリデーション済みの `aws_cli_args` のみを実行する
- shell 文字列の連結実行は採用しない

## 15. 段階的な開発計画

- planner schema

- 読み取り系コマンド生成
- profile / region 認識
- plan 表示

- 確認フロー
- リスク分類の強化
- 更新系操作の一部対応
- エラー要約と修正提案

### Phase 3: Workflow Expansion

- 複数ステップ要求
- 対話的な追加確認
- チームポリシー
- セッション記憶
